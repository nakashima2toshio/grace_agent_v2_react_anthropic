# 用語の前提

この依頼でいう「React化」は、JavaScript/TypeScriptのReactを利用した
Web UIへの移行を意味します。

AIエージェントの思考方式である「ReAct（Reasoning + Acting）」への
アルゴリズム変更を意味するものではありません。

ただし、既存の `agent_support_example.py` がReAct方式を採用している場合は、
その処理を削除・簡略化せず、既存仕様として維持してください。

# 基本方針

## 1. 現有資産を有効活用する

既存コードを十分に調査し、利用できる処理を新規実装し直さないでください。

特に、以下を優先してください。

- `agent_support_example.py` の処理フローを維持する
- 既存のサービス、クラス、関数、設定、プロンプト、型定義を再利用する
- 既存のAnthropicクライアント生成処理を再利用する
- 既存のPlan／Execute／Groundedness／Web検証／Action処理を再利用する
- 既存のテスト、fixture、mock、ドキュメントを再利用する
- CLI固有の入出力と、業務ロジックを分離する
- Reactコンポーネント内へ業務ロジックを持ち込まない
- LLM処理をブラウザから直接呼び出さない
- APIキーなどの秘密情報をReact側へ公開しない
- 移行に不要な既存コードの全面的な書き換えは行わない

既存コードにCLI入出力と業務ロジックが混在している場合は、
まず副作用の少ないサービス層へ抽出し、CLIとWeb APIの両方から
利用できる構造を提案してください。

## 2. 既存コードを読んでから判断する

ファイル名や関数名だけで推測せず、実際の呼び出し経路を追跡してください。

最低限、次の内容を調査してください。

- `agent_support_example.py` のエントリーポイント
- CLI引数と環境変数
- 入力データの形式
- Plan生成処理
- `executor.execute` の定義と呼び出し経路
- Groundedness評価処理
- 回答ゲートと強制エスカレーション条件
- 「情報なし」の検知条件
- Web相互検証の実装
- Action生成処理
- HITL確認処理
- Anthropicクライアントの生成場所
- 使用モデルとモデル設定
- プロンプトの保存場所
- 状態管理方法
- 例外処理、リトライ、タイムアウト、キャンセル処理
- ログ、トレース、利用量・コスト記録
- テストコード
- 既存ReactフロントエンドとバックエンドAPIの構成
- 認証・認可の有無
- ストリーミング通信の有無

# 維持する処理フロー

React化後も、原則として以下の処理順序と判定を維持してください。

1. Plan
2. Execute（`executor.execute`）
3. Groundedness
4. 回答ゲート／強制エスカレーション
4'. 情報なし検知
5. Web相互検証
6. Action（HITL CONFIRM）

各ステップについて、以下を整理してください。

- 入力
- 出力
- 実行主体となる既存クラス／関数
- 状態遷移
- 成功条件
- 失敗条件
- 次のステップへ進む条件
- エスカレーション条件
- ユーザー確認が必要になる条件
- 再実行可能性
- タイムアウト、キャンセル、リトライ
- React画面へ表示する情報
- ログへ記録する情報

処理の実態が上記の説明と異なる場合、既存コードを正として黙って変更せず、
次の形式で差分を報告してください。

- 依頼文上の想定
- 実コード上の動作
- 差分
- 推奨する扱い
- 判断に必要な確認事項

# 期待するWebアプリケーション像

CLIで一括実行されている処理を、React画面から開始し、
各処理ステップの状態と結果を確認できるようにします。

最低限、以下のユーザーフローを検討してください。

1. ユーザーが問い合わせ内容と必要な条件を入力する
2. エージェント実行を開始する
3. Planが表示される
4. Executeの進捗と結果が表示される
5. Groundedness評価結果が表示される
6. 回答ゲートの判定結果が表示される
7. 情報なし検知または強制エスカレーションが表示される
8. 必要な場合、Web相互検証の結果と出典が表示される
9. Action候補が表示される
10. ユーザーがHITL確認画面で承認または却下する
11. 承認後にだけActionを実行する
12. 最終回答、実行結果、エスカレーション結果を表示する

# HITLに関する必須要件

HITL CONFIRMは、単なる確認表示ではなく、Action実行を制御する
安全境界として設計してください。

以下をTODOと設計案へ含めてください。

- 承認前にはActionを実行しない
- 承認、却下、修正依頼を区別する
- 実行予定Actionの内容と影響範囲を表示する
- 外部システムへ送信される情報を表示する
- 二重送信を防ぐ
- 再読み込み後も承認状態を復元できるか検討する
- 承認者、承認日時、対象Action、実行結果を監査ログへ残す
- 古い承認で変更後のActionを実行できないようにする
- 承認APIとAction実行APIの認可を検討する
- 失敗時の再実行方法を定義する
